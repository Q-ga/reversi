# 0001. 盤・石の描画に three.js（WebGL）を採用する

- Status: Accepted
- Date: 2026-06-03

## Context

初期実装は盤・石をCSS（グラデーション＋DOM div）で描いていた。
「石が碁石っぽい」「めくりが一瞬で安っぽい」「全体にショボい」という品質不満が出た。
CSSの3D回転は親要素に perspective が無く平面的な横潰れにしかならず、本物の石が
厚みを持って回転して返る表現に届かない。これはCSS/連番画像では超えられない天井だった。

検討した代替：
- CSS強化＋テクスチャ：軽いが、立体的なめくり・反射の天井が低い。実際に作って不十分と確認。
- 連番画像（スプライト）でのめくり：角度・光の自由度が低く、容量も増える。
- three.js（WebGL）での3D：厚みのある石が物理的に回転し、光源・反射・影が付く。

## Decision

盤・石の描画を three.js による3Dに置き換える。
- 盤：ユーザー提供の盤画像（緑レザー＋金グリッド＋金枠）を真上見下ろしのフラットな面に貼る。
- 石：厚みのある面取り円柱（黒＝光沢／白＝半マット）。環境マップで映り込み、ソフトシャドウ。
- 純粋ロジック層（rules/game/evaluate/notation/stats/events/match）は一切変更しない。描画・演出・音の層だけ差し替える。
- 2D描画のフォールバックは持たない（個人用＝モダンSafari前提）。WebGL非対応環境には案内文のみ表示。
- three.js はオフライン維持のためローカルに同梱し、service worker でキャッシュする（CDN直リンクにしない）。

## Consequences

- 良い面：立体的なめくり・反射・影で品質不満を根本解決できる。ロジックは無傷なので回帰リスクが低い。
- コスト：three.js ぶんの容量増（数百KB）。描画・演出・音は手動／ヘッドレス検証が中心（純粋ロジックのような単体テストは効きにくい）。
- 後戻り：描画エンジンの選択は後から変えにくい。だが描画層が独立しているため、最悪ロジックは再利用してCSS版へ戻せる。
